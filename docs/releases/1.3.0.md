# ModelSync 1.3.0 - CLI, Dry-Run and Migration Reporting

ModelSync 1.3.0 is about making database changes easier to trust. Instead of jumping straight from SQL files to production, you can validate script folders, preview pending work with the CLI, run migrations deliberately, and keep Markdown/JSON reports as deployment evidence. Existing 1.2.x migration runner and provider APIs remain source-compatible.

## Added

- `MigrationExecutionMarkdownReport` for human-readable deployment reports.
- `MigrationExecutionJsonReport` for CI systems, dashboards and structured release evidence.
- Provider support matrix with production and preview provider status.
- Migration reporting guide.
- CLI foundation with `modelsync version`, `modelsync validate`, `modelsync run --dry-run`, and report-writing `modelsync run`.
- CLI mutation requires `--apply`, supports connection strings through `--connection-env`, redacts common secret patterns, and propagates Ctrl+C cancellation.
- Direct CLI argument parsing and secret-handling tests.
- DB-first scaffolder roadmap.
- Migration report, CLI quickstart, and GitHub Actions examples.
- Performance smoke coverage for large migration execution report rendering.
- Opt-in live scale coverage for one million rows on SQL Server, MySQL, MariaDB, PostgreSQL, SQLite, and Oracle.

## Changed

- Oracle is explicitly documented as preview until the public NuGet package ID can be published with package-owner API-key permission.
- NuGet package-set validation and partial-publication recovery cover all eight 1.3.0 packages, including CLI and Oracle.
- Documentation now positions ModelSync as a schema lifecycle toolkit rather than only a DDL generator.

## Fixed

- MySQL 8 check-constraint introspection now resolves table metadata through `table_constraints`.
- PostgreSQL and Oracle default timestamp catalog representations no longer create false destructive type-drift plans; explicit timestamp precision changes are still reported.

## Compatibility

No source breaking change is intended for 1.2.x consumers. Existing migration runner, model synchronizer, stored procedure synchronizer, generator and analyzer APIs remain source-compatible.

## Why It Matters

Production migration tools need more than "it worked" or "it failed". A deployment should be able to answer:

1. Which scripts were applied?
2. Which scripts were skipped?
3. Which script and batch failed?
4. Was a lock acquired?
5. Was history written?
6. How long did each item take?

1.3.0 adds that evidence and a safe CLI entry point without adding another migration engine or a DbUp dependency. ModelSync still uses its own migration runner internally; the CLI is just a cleaner way to use it from terminals and CI.
