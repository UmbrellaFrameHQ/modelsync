# ModelSync 1.4.0 - Migration Execution Hardening

ModelSync 1.4.0 promotes the verified 1.4.0 release candidate to a stable release. It focuses on migration correctness, observable failures, and safer production operation without introducing a new migration engine.

## Fixed

- `RunAsync()` no longer reports a failed batch as successfully applied.
- Changed table scripts no longer adopt the complete target hash after a partial additive repair.
- Dry-run output separates source SQL, executable SQL, repair suggestions, and unapplied drift.
- CLI cancellation maps to exit code `130`.
- SQL Server history comparison no longer treats permission and unrelated missing-object errors as absent history.
- SQL-aware hashing and failed-batch previews handle comments and provider literal forms without exposing complete literal values.
- Analyzer table-model rules recognize `[PostgresTableName]` correctly and no longer raise a false missing-table warning for PostgreSQL models.

## Changed

- SQL Server creates only the configured history schema by default. The former application schema set is opt-in through `LegacyApplicationSchemas`.
- SQL Server and PostgreSQL execute compatible script batches and history writes in the same transaction.
- Execution results distinguish committed, non-transactional, rolled-back, partially applied, cancelled, lock-timeout, and failed outcomes.
- Database reset requires structured approval, expected database validation, and readiness checks. Provider backup support is validated before reset.

## Compatibility

Documented 1.3.0 constructors and migration APIs remain source-compatible. The main behavior changes are failure propagation from `RunAsync()`, manual review for changed table scripts, and removal of application-specific SQL Server schema defaults.

## Migration notes

Read the [1.3.0 to 1.4.0 guide](../migrations/1.3.0-to-1.4.0.md) before upgrading production applications.

## Provider validation

The stable package set passed release build, non-integration tests, live provider integration, repository checks, package-smoke tests, prerelease publication, and an independent public NuGet consumer build. Exact test counts remain in CI evidence because they change as hardening coverage grows.

## Known limitations

- Oracle remains a preview provider and does not provide the stable migration-runner feature set.
- MySQL/MariaDB DDL can commit implicitly; ModelSync does not claim full migration/history atomicity for those operations.
- Table-script parsing is intentionally best-effort. Repair output is review guidance, not proof that all drift is resolved.
- Application-authored migration SQL is trusted input and is not treated as a complete SQL security parser.
- The CLI bundles supported provider clients and cross-platform SQLite native assets, so it is larger than individual provider libraries.
