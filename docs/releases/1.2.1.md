# ModelSync 1.2.1 - Provider API Clarity and Operational Hardening

Release date: 2026-06-30

## Summary

ModelSync 1.2.1 clarifies the public provider API and hardens operational migration execution. It keeps 1.2.0 source compatibility while adding provider-specific default, check and index attributes, structured provider error metadata, failed batch reporting and same-session script execution.

## Why this release was needed

1.2.0 proved the migration runner and live provider model sync flow, but real migration usage showed three practical gaps: provider-specific defaults were not obvious enough in examples, provider database errors lost important diagnostic metadata, and multi-batch SQL scripts needed stronger session continuity guarantees.

## Added

- Provider-specific default attributes.
- Provider-specific raw default SQL attributes.
- Provider-specific check attributes.
- Provider-specific index attributes.
- Typed provider default expression enums.
- Structured provider error metadata.
- Failed batch reporting.
- Consumer compatibility verification.
- Release documentation contract checks.

## Changed

- Canonical examples now prefer provider-specific attributes.
- Core/provider responsibility boundaries are enforced by repository checks.
- Migration batches for a script run in one provider execution scope.
- SQL Server legacy routine normalization is owned by the SQL Server provider.

## Fixed

- Ambiguous Core-first default examples now have provider-specific replacements.
- Database exception number/state/severity/line/object metadata is preserved safely.
- Failed batch index and bounded preview are visible in execution result metadata.
- SQL Server `GO` batches no longer break same-session script expectations.
- Terminal `GO`, SET-option batches and deployment-style SQL Server routine files are normalized when legacy profile is enabled.
- Release and migration documentation is now standardized.

## Deprecated

`DbColumnDefault`, `DbColumnCheck` and `DbColumnIndex` remain supported 1.x compatibility APIs. New code should prefer provider-specific attributes. No compiler or analyzer deprecation warning is introduced in 1.2.1.

## Removed

None.

## Security

- Connection strings, passwords, tokens and API keys are redacted from public results.
- Failed batch preview is capped and normalized.
- Full SQL scripts and raw `Exception.ToString()` output are not copied to public result objects.

## Core/provider responsibility model

Core owns execution lifecycle, planning, policy, history sequencing and provider-neutral result contracts. Provider packages own connection/session behavior, exception extraction, SQL execution, batch splitting, legacy normalization and provider-specific diagnostics.

## Provider-specific API examples

```csharp
[SqlServerColumnDefault(SqlServerDefaultExpression.NewId)]
[SqlServerColumnDefaultSql("CONVERT(uniqueidentifier, SESSION_CONTEXT(N'TenantId'))")]
[SqlServerColumnCheck("Price >= 0")]
[SqlServerColumnIndex("IX_Products_Name")]
```

## SQL Server impact

SQL Server gains typed default expressions, raw default SQL, provider-specific check/index attributes, structured `SqlException` metadata extraction and SQL Server-only legacy routine normalization.

## MySQL/MariaDB impact

MySQL and MariaDB gain typed defaults such as UUID and current timestamp, raw default SQL, check/index attributes and structured provider error extraction.

## PostgreSQL impact

PostgreSQL gains typed defaults such as `gen_random_uuid()`, `CURRENT_TIMESTAMP` and `now()`, raw default SQL, check/index attributes and structured provider error extraction.

## SQLite impact

SQLite gains typed date/time defaults, raw default SQL, check/index attributes and structured SQLite error extraction. Stored procedures remain unsupported by provider capability.

## Legacy migration runner impact

Existing scripts and history continue to work. SQL Server legacy embedded routine files can use terminal `GO`, SET-option batches and trailing deployment noise when `LegacyEmbeddedSql` is enabled.

## Compatibility

1.2.0 consumer source compiles against 1.2.1 candidate packages with zero new warnings and `TreatWarningsAsErrors` enabled. The compatibility proof uses packages only, no project references.

## Migration required

No migration is required. Provider-specific attributes are recommended for new code and can be adopted gradually.

## Known limitations

Composite indexes, included columns, filtered indexes, expression indexes, sort direction and clustered/non-clustered index selection remain out of scope for 1.2.1.
