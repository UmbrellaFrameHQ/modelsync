# ModelSync 1.2.0 to 1.2.1 Migration Guide

## Migration required

No migration is required. Existing 1.2.0 source code continues to compile and run on 1.2.1.

The legacy Core attributes remain supported without new warnings:

- `DbColumnDefault`
- `DbColumnCheck`
- `DbColumnIndex`

New canonical code should prefer provider-specific attributes because they make provider behavior explicit.

## SQL Server

Before:

```csharp
[DbColumnDefault("NEWID()")]
public Guid Id { get; set; }
```

After:

```csharp
[SqlServerColumnDefault(SqlServerDefaultExpression.NewId)]
public Guid Id { get; set; }
```

Raw SQL remains available when the value is truly application-specific:

```csharp
[SqlServerColumnDefaultSql("CONVERT(uniqueidentifier, SESSION_CONTEXT(N'TenantId'))")]
public Guid TenantId { get; set; }
```

## MySQL/MariaDB

Before:

```csharp
[DbColumnDefault("UUID()")]
public string PublicId { get; set; }
```

After:

```csharp
[MySqlColumnDefault(MySqlDefaultExpression.Uuid)]
public string PublicId { get; set; }
```

Raw SQL:

```csharp
[MySqlColumnDefaultSql("CURRENT_TIMESTAMP")]
public DateTime CreatedAt { get; set; }
```

## PostgreSQL

Before:

```csharp
[DbColumnDefault("gen_random_uuid()")]
public Guid Id { get; set; }
```

After:

```csharp
[PostgresColumnDefault(PostgresDefaultExpression.GenRandomUuid)]
public Guid Id { get; set; }
```

Raw SQL:

```csharp
[PostgresColumnDefaultSql("now()")]
public DateTime CreatedAt { get; set; }
```

## SQLite

Before:

```csharp
[DbColumnDefault("CURRENT_TIMESTAMP")]
public string CreatedAt { get; set; }
```

After:

```csharp
[SQLiteColumnDefault(SQLiteDefaultExpression.CurrentTimestamp)]
public string CreatedAt { get; set; }
```

Raw SQL:

```csharp
[SQLiteColumnDefaultSql("CURRENT_DATE")]
public string CreatedDate { get; set; }
```

## Migration checklist

1. Update provider package versions to `1.2.1`.
2. Verify provider `using` namespaces.
3. Move ambiguous Core attributes to provider-specific equivalents where practical.
4. Prefer typed expressions over raw strings when a typed expression exists.
5. Build with `TreatWarningsAsErrors`.
6. Run a compare plan before applying migrations.
7. Run migrations against a test database.
8. Verify migration history tables before production.
9. Record a rollback plan.

## Compatibility notes

- Existing 1.2.0 source compiles.
- No new warnings are expected for legacy consumers.
- `TreatWarningsAsErrors` passes in compatibility validation.
- Use a gradual migration strategy; do not rewrite stable schema models just for style.
