# Migrating from ModelSync 1.3.0 to 1.4.0

ModelSync 1.4.0 is source-compatible with the documented 1.3.0 APIs, but it intentionally changes several migration-runner behaviors. Validate the upgrade against a copy of the target database before deploying it to production.

## RunAsync failures

`RunAsync()` no longer discards a failed execution result. A failed batch, blocked plan, or history failure throws `MigrationExecutionException`. Code that intentionally inspected only returned plans should either handle this exception or use `RunWithResultAsync()` and inspect its structured state.

## Changed table scripts

Changed table scripts now require manual review by default. Do not edit an applied table migration and expect ModelSync to silently converge the database. A changed table script is blocked for manual review, is not executed automatically, and does not record the complete new script hash automatically. If `AutoAddMissingColumnsFromTableScripts` is deliberately enabled, the plan may include additive `RepairSql` suggestions; those suggestions are review-only. The plan is marked `RequiresManualReview` because a best-effort column parser cannot prove that type, default, check, index, or constraint drift was resolved.

`AutoAddMissingColumnsFromTableScripts` is disabled by default. Treat it as a repair-review aid, not as automatic migration convergence.

Deployment tooling should treat these fields separately:

```text
SourceSql            Registered source artifact
PlannedExecutionSql  SQL selected for automatic execution
RepairSql            Review-only additive suggestions
UnappliedDrift       Differences not proven to be resolved
```

Create a new migration for every reviewed change.

## SQL Server schema defaults

The SQL Server runner now adds only its configured history schema by default. Applications that relied on the former `app`, `ref`, `auth`, `log`, `crm`, `exp`, `veh`, and `fin` defaults must configure their schemas explicitly:

```csharp
var options = MigrationRunnerOptions.Default();
options.Schemas.Add("app");
options.Schemas.Add("crm");
```

For a temporary compatibility bridge:

```csharp
options.ApplyCompatibilityProfile(
    MigrationCompatibilityProfiles.LegacyApplicationSchemas);
```

## CLI cancellation

The CLI maps a cancelled run to process exit code `130`. Migration failures remain `1`, invalid usage remains `2`, and successful execution remains `0`.

## Execution states

`Committed`, `CompletedWithoutTransaction`, `RolledBack`, and `PartiallyApplied` describe observed execution outcomes. SQL Server and PostgreSQL use a shared transaction for a compatible script and its history write. MySQL/MariaDB DDL can commit implicitly, so full schema/history atomicity is not claimed there.

## Report serialization

JSON and Markdown reports contain additional root failure, transaction, rollback, and sanitized batch-preview fields. Consumers that deserialize reports into strict closed schemas should allow these additive fields.
